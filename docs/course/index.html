Understanding Esper

Esper is a bare-metal reasoning engine that learns the way a living system adapts: no backpropagation, no hand-installed rules, two clocks running at once — one that adapts in seconds to what's in front of it, one that changes slowly across everything it has ever seen. This course builds that idea up from a single equation to the engine's current research frontier, one short lesson at a time.

Each lesson is numbered module.lesson and assumes only what came before it — skim the sidebar to jump around, or read straight through. Colored tags mark ideas that belong to Esper's two timescales: slow persistent, meta-learned, changes rarely, and fast in-context, adapted per problem, changes constantly. That duality is the spine of everything below.

01

Learning as memory

Esper's architecture is a direct application of a 2025 theory paper, Nested Learning: The Illusion of Deep Learning Architecture (Behrouz, Razaviyayn, Zhong, Mirrokni — Google Research). Its central claim: a deep model isn't one architecture trained by one optimizer — it's a stack of optimization problems running at different speeds. This module distills that theory before any of it touches code.

1.1

The one atom: associative memory

Everything in this theory reduces to one operator. An associative memory M maps a set of keys K to values V, found by minimizing how badly it maps one onto the other: M* = argmin_M L̂(M(K); V). Keys and values don't have to be tokens — they can be gradients, sub-sequences, error signals, anything with a measurable mapping quality.

Definition

"Memory is a neural update caused by an input; learning is the process of acquiring effective memory." — Behrouz et al., §3.1

Memorization is forming any mapping. Learning is forming one that generalizes. That distinction — memorizing versus learning an effective mapping — is the exact line Esper draws between fitting a task's examples and solving the held-out one (Module 3.5).

1.2

Nested systems

A nested system is an ordered stack of levels, each one an optimization problem minimizing its own objective over its own parameters and its own slice of context. The ordering isn't depth in the usual sense — it's update frequency. A higher level updates less often than the one below it.

When every level in the stack happens to be an associative memory (1.1), the paper calls the whole thing an NSAM — a nested system of associative memories. Its claim is that modern architectures, plus the optimizers that train them, are already instances of this pattern; they just don't advertise it as one.

Two design choices define any one module in this frame: (1) what optimization problems it runs and at what frequencies, and (2) how knowledge moves between its levels — directly, by backprop, by meta-learned initialization, or by generating one level's weights from another.

1.3

Optimizers are themselves memories

This is the paper's sharpest reframe. Training a network by backprop is a compression process — each layer is an associative memory mapping its input to its local error signal. And the momentum term in SGD isn't a bookkeeping trick; it's an associative memory that compresses past gradients into its own state. Gradient descent with momentum is already a 2-level nested system: the inner level learns momentum, the outer level uses it. Adam, RMSProp, AdaGrad, Shampoo are all instances of the same pattern with a richer inner memory.

Standard momentum is a weak low-pass filter — at β = 0.9, the last ~43 gradients hold over 99% of the weight — which is why it forgets the gradient subspace of an earlier, orthogonal task. Every mitigation for catastrophic forgetting later in this course is, underneath, a fix to this same forgetting mechanism.

1.4

The Continuum Memory System

north star

Drop the binary of "short-term vs. long-term memory." Replace it with a Continuum Memory System (CMS): a chain of memory blocks, each updated at its own frequency C^(ℓ). High-frequency blocks behave like short-term, fast adaptation; low-frequency blocks behave like persistent, slowly-won knowledge. A Transformer's MLP block is just the special case of one block, one frequency.

Fig 1.4 — CMS as a spectrum of blocks, not a two-tier cache

The payoff is resistance to forgetting: when one block gets overwritten by new context, older knowledge still lives undisturbed in the other-frequency blocks, and can be recovered through cross-level transfer. Blocks only compute on their own schedule, so updates stay cheap and parallelizable.

1.5

Self-modifying memory, and HOPE

A standard Transformer's key/query/value projections are fixed after training — whatever contextualization happens, it's bounded by that first, frozen level. A self-modifying memory breaks that: it generates its own keys, values, learning rate η, and retention gate α, per input, and optimizes itself against an internal objective. It is, in a real sense, controlling its own learning process.

HOPE

Self-modifying Titans followed by a CMS chain (1.4): a small, richly-adaptive block handles fast, in-context change; a multi-frequency stack behind it holds persistent knowledge. Small capacity + rich rule, feeding large capacity + simple rule.

1.6

Terms this reframes

  • In-context learning isn't an emergent surprise — it's the direct consequence of having multiple frequency levels at all. Any level adapting to its context is doing ICL, by definition.
  • Pre-training is just the lowest-frequency level: ICL where the context happens to be the entire dataset.
  • Test-time training is parametric in-context learning whose acquired knowledge vanishes the moment the context is removed.
  • There is no train/test boundary inside a neural learning module — only two states: receiving input, or sitting isolated.

One more consequence worth carrying forward: momentum state, optimizer state, RNN hidden state — these are all knowledge stores too, not just the weights that backprop touches. Module 3 will make the same move for Esper: its "parameters" include a fast-changing part and a slow-changing part, and both are learned.

02

Evolution strategies

Nested Learning describes optimizers as memories in the abstract. Esper needs an actual one — and it can't be backprop. This module covers the mechanism that fits every memory in the rest of the course: a derivative-free Evolution Strategy (ES).

2.1

Why not backprop

Esper is built under a hard constraint: zero external ML libraries, and no autodiff framework in the runtime path. That's not an implementation detail, it's a methodological stance — nothing about how the engine learns should be inherited from a framework unexamined. So gradients aren't computed by backpropagating through a computation graph; they're estimated by perturbing parameters and observing how a scalar fitness responds.

2.2

Antithetic sampling

One ES step: draw real Gaussian noise ε, evaluate the fitness function F at two mirrored points, w + σε and w − σε, and combine the two scores into a gradient estimate. Because the pair is symmetric around w, the noise on either side cancels in expectation and what's left points toward the direction that actually improved fitness.

Fig 2.2 — a mirrored pair straddling the current parameter, on a 1-D slice of fitness
graph LR
  N["draw noise ε ~ N(0,1)"] --> P["w + σ·scale·ε"]
  N --> Q["w − σ·scale·ε"]
  P --> FP["F(w+σε)"]
  Q --> FM["F(w−σε)"]
  FP --> G["grad += (F+ − F−) · ε"]
  FM --> G
  G --> U["w += (α / 2Nσ) · scale · grad"]
Fig 2.2b — one antithetic ES iteration, averaged over N pairs

scale is a per-parameter preconditioner — different parameter groups (say, geometry versus colour) can need different step sizes, so one shared σ gets rescaled per group rather than forcing a single compromise.

2.3

Anneal: explore wide, then settle sharp

A single fixed σ can't do both jobs a fit needs. Early on, the search needs to be wide enough to discover a qualitatively different region of parameter space (a large layout change like a transpose). Late in the fit, it needs to be narrow enough to settle onto an exact value without noise pushing it back off. Esper anneals both σ and the step size α across the fit — wide and fast at the start, small and cautious by the end — over a few thousand iterations.

Fig 2.3 — the annealed sigma schedule: explore, then settle onto exact values
2.4

Two timescales, and the slow one is meta-learned

slowfast

Here the ES core stops being single-task. An outer, low-frequency loop — a Reptile meta-loop — runs across many tasks: for each one, it fits fast in-context starting from the current slow, then nudges slow a small step toward wherever fast ended up: slow += META_LR · (fast − slow). 

graph TD
  S["slow — the prior
(persists across tasks)"] -->|"fit_operator, anchored to slow"| F["fast — per-task fit
(in-context, from demos)"]
  F -->|"reptile: slow += META_LR·(fast − slow)"| S
Fig 2.4 — the outer Reptile loop turning a fixed prior into a meta-learned one

This is the Nested-Learning idea from 1.2, made concrete: two optimization problems at two frequencies, with knowledge transferred between them by initialization — exactly the MAML-style transfer the paper names. A fresh, in-family task can now be fit at a narrow, cheap budget where a cold (identity) prior simply fails: warm start plus a cheap local fit beats cold start plus a wide search, but only because the prior has already absorbed the family's shape.

03

Esper's core architecture

With the theory (Module 1) and the optimizer (Module 2) in hand, this module opens the engine itself: how it stores a task, what its two weight sets actually are, and the seam that lets new memory types be added without touching the learning core.

3.1

The arena: memory with no allocator in the hot loop

Esper is pure Mojo, bare metal, no dynamic allocation inside a hot loop. Its arena (HopeArena) is a move-only bump-pointer allocator: a contiguous block of memory where allocation is "wherever the pointer currently is," and the pointer only ever moves forward. Nodes placed into it (HopeNode) are kept strictly POD — raw pointers and integers, no owning members — specifically so they can be moved into bump-allocated memory without a per-node heap allocation.

Fig 3.1 — the arena: allocation is a pointer bump, never a free list
3.2

fast and slow, mapped onto HOPE

slowfast

Every HopeNode carries two weight slices: slow and fast. This is Module 1's two-timescale idea, given a name in code. fast is fit fresh in-context on one task's demonstration pairs; slow is the meta-learned prior it's anchored to, updated only by the outer Reptile loop (2.4) across many tasks.

3.3

The seams: Domain and Memory

Two traits keep the ES core generic. A Domain defines what a problem looks like — an example type, a distance metric, a score — without saying anything about grids specifically (a grid domain and a 1-D sequence domain both conform, with zero change to the ES). A Memory defines what gets fit: a parameter count, a seed, and an apply function from weights + input to output. 

graph TD
  Dom["Domain trait
Example type · distance · score"] -->|associated type| Mem["Memory trait
param_dim · seed · apply(weights, input) → output"]
  Mem --> Fit["ES core: fitness / evolve_fast_weights / fit_operator"]
  Fit --> Score["held-out score, via Domain.distance"]
Fig 3.3 — the seam the ES core reaches every problem through

Crucially, there is no runtime memory-selector — which memory to use is a compile-time choice, not a dispatch decision the engine makes at runtime. A runtime selector over hand-written memories would itself be a symbolic DSL by another name, which is exactly what the project's first value forbids (Module 7.1). New memory types are added additively, and each is measured only on the subset of problems it can express.

3.4

Worked example: the structured operator

The first working Memory is a hand-structured one, useful precisely because it's simple enough to read end-to-end. Its 16 parameters are a 6-value centered affine transform (the geometry: flips, transpose) plus a 10-entry colour lookup table. It runs colour, then gather: each of the four corner input cells is mapped through the colour table before being blended by the geometry, which decouples the colour fit from the geometry's precision. The gather is smooth (bilinear) so the ES always has a usable gradient signal, yet lands exactly on the target at integer parameter values.

Why this memory is later retired

Its 16 hand-picked parameters can only express affine geometry plus a global recolor. Module 4 replaces it, piece by piece, with memories that learn the same behavior — and more — with nothing hand-installed.

3.5

The metric that can't be cheated

Every memory is fit on a task's demonstration pairs, then scored on an unseen test pair from the same task. Fitting the demos well proves nothing by itself — a memory with enough free parameters can memorize any finite set of examples. Generalizing to the held-out pair is the only claim that a genuine transformation, not a lookup table, was found.

The cold-fit bar

If a memory only clears this bar via per-task hand-staging, mid-fit parameter boosts, or bespoke tuning, the project calls that "stone soup": the reasoning lived in the human who staged it, not in the model. Such results are recorded as documented negative results, never as milestones.

04

The emergent memory zoo

This is the project's longest-running thread: replacing every hand-coded piece of the structured operator (3.4) with something the engine learns for itself, one training wheel at a time, each new memory measured honestly on what it can and can't express.

4.1

First emergence: a per-cell MLP

fast

The first training wheel to come off is the colour lookup table. MLPMemory is a tiny per-cell 1→H→1 tanh network, fit the same way as everything else — by ES, on the demos. It learns to recolor a grid to held-out 1.0 with no lookup table hand-installed: the mapping itself is discovered, not looked up.

4.2

Emergent geometry: attention as a gather

AttnGatherMemory replaces the hand-coded affine transform with a 7-parameter learned position-attention gather: a 2×2 coordinate projection plus a temperature, softmax-weighted over a small window. With integer-valued projections it reproduces flip/flip/transpose exactly — but nothing about "flip" or "transpose" was ever named in the code. Geometry became a special case of attention, learned from scratch, first try.

4.3

The composition pattern

Naively fitting one memory that does both geometry and colour hits a structural wall: geometry needs a soft gather to have a usable gradient, colour needs sharp bins to stay clean, and one shared temperature can't serve both — a free colour table will quietly absorb geometry error at the wrong position, flattening the very signal the ES needs.

Negative result, recorded

A single combined geometry+colour memory was tried and abandoned — it only reached the bar under hand-staging (stone soup, 3.5). The fix was a different factorization, not more tuning.

The recipe that replaced it, now proven three times over: find a representation where the two factors commute, fit each factor on a signal the other one cannot touch, then compose forward. Concretely — the colour table is written closed-form from count signatures that are position-free (so geometry can't corrupt them), and the geometry ES then runs on demos already pre-mapped through that fixed colour table.

Fig 4.3 — a commuting square: colour-then-geometry equals geometry-then-colour

This composed memory (GeomColorComposedMemory) emergently retires the hand-coded structured operator from 3.4: it matches its whole expressible subset held-out 1.0, cold, and additionally solves a composed flip∘recolor task no single memory in the zoo could express alone.

4.4

Self-modifying memories

fast

A derivative-free ES cannot search a large, coupled fast-weight space directly — there just aren't enough antithetic samples in the world to find a high-dimensional needle by random perturbation. The fix, straight out of Module 1.5: move fast adaptation off the ES entirely. The memory writes its own fast state from the demos in a single forward pass — a self-write rule — and the ES only ever searches the small, slow vector that shapes how that rule reads.

The delta rule

The self-write updates a state matrix by S ← (1−α)S + η(v − S·k)k, with key k, value v, learning rate η, and retention gate α all self-generated by the memory. On a fresh, arbitrary sequence rule, meta-learning these projections cold lets a new task adapt in one pass to ~0.98 held-out — no per-task ES fit at all.

4.5

Self-mod on a grid: three expressivity barriers

Lifting self-write to 2-D grids happened in three steps, each crossing a barrier the last one provably couldn't:

  • Context keys — key on a centre cell combined with its neighbours (an outer product), read linearly. Expresses additive center↔neighbour rules.
  • Neighbourhood histograms — key on a centre-free count of matching Moore-8 neighbours, read through a sigmoid threshold. A linear read provably cannot express this disjunctive/count class; the nonlinearity is load-bearing, not decoration (measured: linear read 0.61, nonlinear 0.995).
  • Count maps — an arbitrary, non-monotone count→colour map. The obvious fix (learn a value table by gradient covariance) fails outright here, because identifying which colour to count is a discrete choice, not something a smooth gradient target can express. The working fix is a meta-learned scoring salience over demo statistics — the model learns to weigh variance-reduction over correlation, the one feature that survives non-monotone maps.
4.6

When output shape is itself unknown

Every memory so far assumed the output grid is the same size as the input. Real tasks crop, subsample, tile, and upscale. The fix is a parallel ShapeMemory seam: the output size is inferred in-context — a per-axis affine rule out = round(k·in + b), written closed-form by least-squares over the demo dimension pairs — then the same attention gather (4.2) reads from the input grid into an output-shaped query. A later addition, a symmetric "fold" read instead of a periodic wrap, made mirror-tiling (kaleidoscope) patterns expressible too — the torus wrap alone provably cannot produce that sawtooth.

4.7

Content-addressed memory

The stubbornest remaining class of tasks writes output where the input evidence is not — fill an enclosed region, extend a ray, copy the nearest object's colour. No per-cell function of position-aligned context can express that; the output has to be fetched from somewhere else on the grid. A GridSubstrate precomputes reusable representations (connected components, nearest-object BFS, ray sweeps) that a written 16-entry relational action table (keep / copy-fetched / constant) reads from — a genuinely content-addressed gather, composed on top of everything built so far.

That's the honest state Module 4 leaves things in — the reason the project's live roadmap keeps naming new "rungs" rather than declaring done.

05

Open-ended mastery

Everything in Module 4 still hands the engine a goal — a task's demonstration pairs are compressed supervision, even without a hand-coded DSL. Vision B asks a stricter question: can the same ES core master an environment with zero hand-coded goals at all? Five rungs, each reward-free, each scored through the unchanged generic ES core from Module 2.

5.1

The last hand-out

A demonstration pair is still a goal someone else chose. Vision B replaces it with intrinsic motivation — self-generated novelty stands in for supervision entirely. This is a stricter reading of the project's founding value than Vision A ever attempted, because it drops the last hand-out: the goal itself.

5.2

A reward-free world

The testbed is a deterministic 16×16 gridworld: an avatar with six actions (move × 4, paint, cycle-brush) under one parameterizable dynamics rule (gravity). There is no reward channel — the only thing the world produces is state. A SandboxDomain/SandboxPolicyMemory pair conforms the same generic fit_operator from Module 2 to score whole rollouts, so the ES core needs zero new code to run in this world at all.

5.3

Novelty search

An NS-ES meta-population's only fitness signal is how novel its own trajectory is against an archive of everything seen so far — never a task score. Against an equal-budget random-policy baseline, directed novelty search covers distinctly more of the world.

Measured
4.9×distinct cells vs. random
3.0×distinct end-states vs. random

A calibration finding worth keeping: raw novelty deltas are tiny and shrink as the archive fills up, so unit-std fitness shaping of the ES step is load-bearing — an unshaped step barely moves anything.

5.4

MAP-Elites: a persistent skill repertoire

Novelty search is transient — it forgets. MAP-Elites fixes that by keeping a permanent map: one slot per distinct end-state "cell," holding the best (most direct) policy ever found for reaching it. Once a skill is stored, it's replayable forever.

Fig 5.4 — each filled cell is one stored, replayable skill, keyed by end-state

5.5

Empowerment, computed exactly

Empowerment measures how much an agent's actions control its future states — normally an intractable estimation problem. But the sandbox is fully deterministic, so the usual channel-capacity algorithm collapses to something exact: n-step empowerment is just log₂(number of distinct states reachable in n steps), computed by brute enumeration. No learned parts, no archive.

Measured

3.4× more elites stored than a random-policy baseline. Caveat: empowerment is a concentrating signal — it needs frequent elite-restarts or it parks below random, since routes that stay optionality-rich are a narrow target.

5.6

A world model, and learning progress

A transition (state, action → next state) is just another ExamplePair, so the same unchanged ES core fits a predictive model of the world's dynamics — no new learning machinery. Learning progress (LP) is then defined honestly: the ES fitness slope, not the raw error. That distinction matters because a region of pure noise (a "noisy TV") has permanently high error but zero slope — LP correctly ignores it, where a naive novelty-seeking agent would get stuck staring at static forever.

Measured
22×LP on novel regions vs. mastered/scrambled
2.4×LP-guided collection vs. uniform (0.362 vs 0.153)
5.7

The convergence test

This is the hypothesis the whole ladder was built to test: do skills discovered with zero goals (Vision B) become a reusable vocabulary that few-shot, goal-driven composition (Vision A) can draw on? A held-out goal is a target end-state the repertoire never stored. Seeding the ES fit with the repertoire's nearest stored skill — real indexed retrieval, not a generic warm-init — is compared against starting cold.

Measured
7.3×closer to goal than cold-start
29.6×closer than a random-elite control

Retrieval, specifically — not just "having a warm start" — is the lever: a random elite barely beats cold, but the nearest one wins by an order of magnitude. A composition head over several frozen retrieved primitives does even better on multi-part goals, though its margin is bounded by how dense the repertoire already is nearby.

5.8

ACCEL: the last hand-coded thing goes too

One thing was still hand-picked across every rung above: the fixed set of training worlds. ACCEL retires it — mutate a level's config, keep a replay buffer curated by learnability (5.6's honest, noise-immune slope), and never hand-write a curriculum. Compared byte-for-byte against domain randomization (a fresh random level every round) at the identical compute budget:

Measured
0.154held-out accuracy, curated curriculum
0.0held-out accuracy, domain randomization

Domain randomization's difficulty swings churn an incremental learner all the way to zero, while curation holds the world in the moderate-difficulty band where learning actually happens. The honest limit: the sandbox's single dynamics rule (gravity) is the ceiling on this — a richer curriculum needs a richer world. That's exactly what Module 6 builds next.

06

The frontier: cross-world transfer

The most recent, active direction. This is where the roadmap is being written as this course is — treat it as the current state of an open question, not a settled result.

6.1

The pivot

A direction change: solving ARC-AGI is treated as less important than learning from diverse problems and worlds — sokoban, tic-tac-toe, tangram, untying knots — and carrying that knowledge from one to the next. This elevates 5.7's convergence hypothesis from one interesting result to the project's measured currency: transfer — what a warm start earns you over a cold one, at equal budget, on the next world, always scored held-out.

6.2

Walls: a second world

Before transfer can be measured, there has to be a second world to transfer into. Walls add a topology constraint to the sandbox (5.2): a reserved in-grid value the avatar can't pass and gravity treats specially. Built as a strict superset of the original world — every one of the five-rung ladder's proofs still passes byte-identically — so world 2 doesn't invalidate world 1, it extends it.

6.3

What carries, what doesn't

A repertoire built entirely in the open world (world 1) is asked to warm-start goals in the walled world (world 2) it has never seen. Three arms: cold-start, a random stored elite, and nearest-neighbour retrieval by behaviour.

Measured — T-POC-1

A randomly chosen world-1 elite is 4–8× worse than starting cold in world 2 — strong negative transfer. BC-nearest retrieval rescues that back to roughly cold-start parity, 5–8× closer than the random-elite control across two seeds, and hits the exact goal in 5/24 cases where cold-start managed ≤2/24.

Read carefully, that's a specific and slightly humbling finding, distilled as: the index carries, the warm basin does not. Knowing which skill is relevant transfers across worlds. The actual learned weights of that skill — tuned for a world without walls — mostly don't; they need to be re-fit in the new world almost from scratch once retrieved.

6.4

Where this points next

If the index is what transfers and the weights don't, the natural next question is what a memory would need to make the weights transfer too — perhaps a representation of a skill that's less tied to one world's specific geometry, or a CMS-style (1.4) persistent slow layer that consolidates what's shared across worlds rather than what's specific to one. This is exactly the "rung #6" question Module 4.8 named for Vision A, now asked at the level of whole worlds instead of individual tasks — the two visions' open threads are converging on the same shape of problem.

07

Wrap-up

A last pass over the rules the whole project runs by, a compact glossary, and where to go for the living, evolving version of everything summarized here.

7.1

The six values

  1. 01Emergence over hand-coding. No symbolic DSL in the runtime path; structural priors are removable training wheels, never the destination.
  2. 02Honest measurement. Held-out generalization is the only currency — uncheatable by memorization. A scaffolded pass is a negative result.
  3. 03Build from bare metal. Pure Mojo, zero external ML libraries, ES instead of backprop — nothing about how it learns is inherited unexamined.
  4. 04Domain-agnostic core. The ES / two-timescale / self-modifying spine stays separable from any one domain — progress must transfer toward the mission, not just move one number.
  5. 05Frugal by design. Every milestone must be provable on commodity hardware with tiny corpora — a claim needing a cluster is a sign the substrate isn't efficient enough yet.
  6. 06Narrated evidence trail. Every direction change carries its why, so the project's turns are auditable, not vibes.
7.2

Glossary

Associative memory
A mapping M from keys to values, fit by minimizing a loss over that mapping. The one atom every other term in this course builds on.
Fast / slow weights
Esper's two timescales: fast is fit in-context per task; slow is the meta-learned prior, updated only by the outer Reptile loop.
Evolution Strategy (ES)
A derivative-free optimizer: estimate a gradient by perturbing parameters and observing fitness, rather than backpropagating through a graph.
Domain / Memory trait
The two seams the ES core is generic over — a problem's shape (Domain) and what gets fit to solve it (Memory) — with no runtime selector between memories.
Held-out generalization
Fit on a task's demonstration pairs, score the unseen test pair. The project's only success metric.
Cold-fit bar / stone soup
A result only holds if it clears the bar with no per-task human staging. A staged pass is a documented negative result, not a milestone.
Composition pattern
Find a commuting representation, fit each factor on a signal invariant to the others, compose forward — the recipe behind every composed memory in Module 4.
Novelty / empowerment
Two reward-free intrinsic signals: how new a state is (novelty), and how much an agent's actions control its future (empowerment).
Learning progress (LP)
The slope of prediction error over time, not the error itself — the noise-immune signal behind guided curriculum collection.
Transfer
What a warm start earns over a cold one, at equal budget, in a new world — the project's current measured currency (Module 6).
